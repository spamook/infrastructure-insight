# Infrastructure Insight

A full-stack infrastructure monitoring application that displays real-time system metrics (hostname, OS, CPU, RAM, uptime) from the server responding to each request.

The application is containerized with Docker and deployed across a multi-server cluster managed by Vagrant.

---

## Reviewer Setup (how to run this project)

### Requirements

- [Vagrant](https://www.vagrantup.com/downloads) installed
- [VirtualBox](https://www.virtualbox.org/wiki/Downloads) installed
- ~6 GB of free RAM (5 VMs x 1 GB each)

### Steps

**1. Create a working folder and clone the repo inside it:**

```bash
mkdir cluster
cd cluster
git clone https://gitea.kood.tech/danilakargajev/infrastructure-insight.git
```

**2. Copy the Vagrantfile one level up (next to the repo folder):**

```bash
cp infrastructure-insight/Vagrantfile ./Vagrantfile
```

The folder structure must look like this:

```
cluster/
  Vagrantfile
  infrastructure-insight/
    backend/
    frontend/
    docker-compose.backend.yml
    ...
```

**3. Start the VMs:**

```bash
vagrant up
```

This will create and provision 5 virtual machines. First boot takes several minutes.

**4. Deploy the application on each server:**

Follow the [Deployment](#deployment) section below, using `vagrant ssh <server-name>` to access each VM.

**5. Open the application:**

```
http://192.168.56.10
```

## Architecture

```
Client
  |
  v
Load Balancer (192.168.56.10)
  nginx — distributes traffic using a resource-based adaptive algorithm
  |              |
  v              v
Web Server 1   Web Server 2       (192.168.56.11 / 192.168.56.12)
  frontend        frontend        React + nginx container (port 80)
  sidecar         sidecar         Backend sidecar container (port 3001)
                                  Exposes host metrics to load balancer
  |              |
  v              v
        App Server (192.168.56.13)
          backend — Node.js/Express container (port 3001)
          Collects and serves system metrics via REST API

        Backup Server (192.168.56.14)
          Runs weekly rsync backups of /home and /etc
          from all servers via SSH
```

## Deployment

### 1. Start the VMs

```bash
cd /path/to/project
vagrant up
```

### 2. Deploy the backend (app-server)

```bash
vagrant ssh app-server
cd /vagrant/infrastructure-insight
docker-compose -f docker-compose.backend.yml up -d --build
```

### 3. Deploy the frontend (web-server-1 and web-server-2)

Run the following on both web servers:

```bash
vagrant ssh web-server-1  # then repeat for web-server-2
cd /vagrant/infrastructure-insight
echo "BACKEND_HOST=192.168.56.13" > .env
docker-compose -f docker-compose.frontend.yml up -d --build
docker-compose -f docker-compose.sidecar.yml up -d --build
```

The sidecar container exposes the host machine's CPU and RAM metrics on port 3001. The load balancer uses these metrics to calculate traffic weights.

### 4. Configure the load balancer (load-balancer)

```bash
vagrant ssh load-balancer
sudo apt install nginx jq -y
sudo cp /vagrant/infrastructure-insight/load-balancer/nginx.conf /etc/nginx/conf.d/lb.conf
sudo nginx -t && sudo systemctl reload nginx
```

Set up the adaptive load balancing script:

```bash
mkdir -p ~/scripts
cp /vagrant/infrastructure-insight/scripts/adaptive_lb.sh ~/scripts/
chmod +x ~/scripts/adaptive_lb.sh

sudo touch /var/log/adaptive_lb.log
sudo chown devops:devops /var/log/adaptive_lb.log

# Allow nginx reload without password
echo "devops ALL=(ALL) NOPASSWD: /usr/sbin/nginx" | sudo tee -a /etc/sudoers

# Run every minute via cron
(crontab -l; echo "* * * * * /home/devops/scripts/adaptive_lb.sh") | crontab -
```

### 5. Configure the backup server

```bash
vagrant ssh backup-server
mkdir -p ~/scripts /backups
cp /vagrant/infrastructure-insight/scripts/backup.sh ~/scripts/
cp /vagrant/infrastructure-insight/scripts/restore.sh ~/scripts/
chmod +x ~/scripts/backup.sh ~/scripts/restore.sh

# Schedule weekly backup every Sunday at 02:00
(crontab -l; echo "0 2 * * 0 /home/devops/scripts/backup.sh") | crontab -
```

## Load Balancing Algorithm

The load balancer uses a **resource-based adaptive algorithm**.

Every minute, the script `adaptive_lb.sh` fetches `/metrics` from both web servers and calculates a load score for each:

```
score = (cpu_load_avg_1min * 50) + ram_usage_%
```

A higher score means the server is more loaded. nginx weights are then assigned inversely proportional to the score (range 1 to 10). The server with lower load receives more traffic.

If a server becomes unreachable, all traffic is routed to the other server automatically.

Example log output:

```
[2026-06-08 17:50:59] scores  -> web1=42.1  web2=87.6
[2026-06-08 17:50:59] weights -> web1=8  web2=2
[2026-06-08 17:50:59] nginx reloaded (web1=8 web2=2)
```

## Backup System

The backup server runs a weekly rsync backup of `/home` and `/etc` from all servers over SSH.

To run a manual backup:

```bash
~/scripts/backup.sh
```

To restore from a backup:

```bash
~/scripts/restore.sh <server-name> <date>
# Example:
~/scripts/restore.sh web-server-1 2026-06-08
```

Backup logs are written to `/var/log/backup.log`.

## Verification

Check that the application is running through the load balancer:

```bash
curl http://192.168.56.10/api/metrics
```

Confirm traffic is distributed between both web servers by checking the hostname field in repeated requests:

```bash
for i in {1..6}; do
  curl -s http://192.168.56.10/api/metrics | python3 -c "import sys,json; print(json.load(sys.stdin)['hostname'])"
done
```

Monitor adaptive load balancing decisions in real time:

```bash
tail -f /var/log/adaptive_lb.log
```
